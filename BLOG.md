# План: «Лента» (блог со статьями)

Документ описывает добавление раздела **Лента** — ленты статей с лайками и комментариями. **Код по этому документу пока не меняется**; это план для последующей реализации.

---

## 1. Цели и требования

| Требование | Решение |
|------------|---------|
| Просмотр статей | Доступен **всем**, в том числе без авторизации |
| Публикация статей | Только **авторизованные** пользователи |
| Лайки | Только **авторизованные**; один лайк от пользователя на статью (переключатель: поставить / снять) |
| Комментарии | Только **авторизованные**; просмотр комментариев — **всем** (вместе со статьёй) |
| Модерация | На первом этапе **не обязательна** (статьи публикуются сразу); ниже — опция для v2 |

**Именование в UI:** в навигации и заголовках использовать **«Лента»**; URL раздела — **`/lenta`** (см. п. 11).

## 3. Модель данных (Prisma)

### 3.1. Сущности

**`FeedPost` (статья)**

- `id` — UUID  
- `authorId` — FK → `User`  
- `title` — string, ограничение длины (например 200)  
- `slug` — уникальный string для ЧПУ (**v1 не обязателен** — URL страницы по `id`; **позже** — slug в маршруте и заполнение поля, см. п. 11)  
- `excerpt` — краткое описание, nullable или обязательное короткое  
- `body` — `String` `@db.Text` (**хранение в Markdown**, рендер на фронте с **санитизацией**, см. п. 11)  
- `coverImageUrl` — nullable string (URL из существующего upload)  
- `status` — enum, минимум: `published` (на v2: `draft`, `moderation`)  
- `publishedAt` — `DateTime` (для сортировки ленты; при мгновенной публикации = `createdAt` или отдельное поле)  
- `createdAt`, `updatedAt`  
- Индексы: `authorId`, `publishedAt` desc, `(status, publishedAt)` для списка  
- **Счётчик просмотров (уникальные пользователи):** не дублировать числом на посте — считать по связанной таблице (см. `FeedPostView` ниже). В API отдавать поле `uniqueViewCount` (или аналог) как результат `COUNT` уникальных зрителей.

**`FeedPostView` (факт просмотра статьи уникальным пользователем)**

- `id` — UUID  
- `postId` — FK → `FeedPost`, onDelete Cascade  
- `viewerId` — FK → `User`, onDelete Cascade (**только авторизованные** учитываются в метрике «уникальные пользователи»)  
- `firstViewedAt` — `DateTime` @default(now()) (или одно поле `viewedAt`)  
- **`@@unique([postId, viewerId])`** — не более одной записи на пару пост + пользователь; повторные открытия страницы не увеличивают счётчик  
- Индекс: `postId` для агрегации  

*Просмотры без авторизации:* по требованию «уникальные пользователи» в счётчик **не входят**; при необходимости позже добавить учёт гостей (например cookie `anon_id` + отдельная таблица или поле nullable `viewerId` + уникальность по `(postId, anonKey)` — вынести в отдельную задачу).

**Регистрация просмотра**

- При открытии страницы статьи авторизованным пользователем: `INSERT ... ON CONFLICT DO NOTHING` (или upsert Prisma) в `feed_post_views`.  
- `GET` детали поста для гостя: счётчик показывать как есть (только залогиненные уникальные), без записи строки.

**`FeedPostLike`**

- `id` — UUID  
- `postId` — FK → `FeedPost`, onDelete Cascade  
- `userId` — FK → `User`, onDelete Cascade  
- **`@@unique([postId, userId])`** — один лайк на пару пост–пользователь  
- `createdAt`  

**`FeedComment`**

- `id` — UUID  
- `postId` — FK → `FeedPost`, onDelete Cascade  
- `authorId` — FK → `User`, onDelete Cascade  
- `body` — `Text`, лимит длины на уровне Zod (например 2000 символов)  
- `createdAt`, `updatedAt` (редактирование **своих** комментариев — **v1**, см. п. 11)  
- Индекс: `postId`, `createdAt`  

**Связи в `User`**

- `feedPosts`, `feedPostLikes`, `feedComments`, `feedPostViews` (имена на усмотрение, главное — не конфликтовать с существующими relation names в Prisma).

### 3.2. Enum (пример)

```prisma

  published
  // draft      // v2

}
```

## 4. API (контракт, черновик)

Общий формат ответов — как в проекте: `{ success, data }` / ошибки с `message`.

### 4.1. Публичные (без JWT)

| Метод | Путь | Назначение |
|-------|------|------------|
| `GET` | `/api/feed/posts` | Список статей: пагинация `page`, `limit`, сортировка по `publishedAt` desc; в ответе — автор (id, name, avatarUrl, companyName), счётчик лайков, **`uniqueViewCount`** (число уникальных авторизованных зрителей), **опционально** `likedByMe: boolean` если передан валидный токен (через `optionalAuthenticate`) |
| `GET` | `/api/feed/posts/:id` | Одна статья + те же поля автора, число лайков, **`uniqueViewCount`**, список комментариев (с автором), при опциональной авторизации — `likedByMe`. Рекомендуется при **авторизованном** запросе зарегистрировать просмотр (`upsert` в `feed_post_views`) в рамках этого же обработчика или отдельным `POST` после загрузки страницы (избегать двойного учёта при SSR + клиент — зафиксировать одну точку вызова). |

Рекомендация: зарегистрировать эти маршруты с **`preHandler: [app.optionalAuthenticate]`**, чтобы в одном запросе отдавать «лайкнул ли текущий пользователь», не дублируя эндпоинты.

### 4.2. Только с JWT (`preHandler: [app.authenticate]`)

| Метод | Путь | Назначение |
|-------|------|------------|
| `POST` | `/api/feed/posts` | Создать статью (валидация Zod) |
| `PUT` | `/api/feed/posts/:id` | Редактировать **свою** статью (`authorId === userId`) **или чужую — только роль `moderator`** |
| `DELETE` | `/api/feed/posts/:id` | Удалить **свою** статью **или чужую — только роль `moderator`** |
| `POST` | `/api/feed/posts/:id/like` | Поставить лайк (идемпотентно: если есть — 200 без ошибки или 409 — выбрать один стиль) |
| `DELETE` | `/api/feed/posts/:id/like` | Снять лайк |
| `POST` | `/api/feed/posts/:id/comments` | Добавить комментарий |
| `PUT` | `/api/feed/comments/:commentId` | Редактировать **свой** комментарий **или чужой — только `moderator`** |
| `DELETE` | `/api/feed/comments/:commentId` | Удалить **свой** комментарий **или чужой — только `moderator`** |

**Права модератора (зафиксировано, п. 11):** **только** пользователь с ролью **`moderator`** может удалять и редактировать **чужие** посты и комментарии. Обычный пользователь — только свои. Проверять `getUserRole(request) === "moderator"` наряду с проверкой автора.

### 4.3. Валидация (Zod)

- Заголовок, тело, excerpt — min/max длины.  
- `coverImageUrl` — опционально `z.string().url()` если URL с бэкенда.  
- Комментарий — не пустой, верхняя граница длины.  
- Защита от XSS: тело статьи — **Markdown** (п. 11); парсинг + **санитизация HTML** на выводе. Комментарии — по политике: обычно plain text с экранированием или тот же подход, если разрешите разметку позже.

### 4.4. Производительность

- Для списка постов: `_count` лайков и **`_count` по `feedPostViews`** (или подзапрос) для `uniqueViewCount`, не тянуть сами лайки/просмотры.  
- Комментарии на странице списка **не** подгружать; только на странице поста с пагинацией (`GET .../posts/:id/comments?page=&limit=` или вложенный массив с лимитом «последние N» на v1).

## 8. Безопасность и злоупотребления

- Rate limiting на создание постов/комментариев — по возможности на уровне прокси (Timeweb) или middleware Fastify (отдельная задача).  
- Запрет HTML в пользовательском вводе или санитизация.  
- CSRF: при текущей схеме JWT в заголовке для API — типичный риск ниже, чем cookie-only.  
- Проверка прав на каждый `PUT`/`DELETE`: автор **или** `moderator` (для чужого контента — только модератор).

## 10. Чек-лист проверки (QA)

### База данных

- [ ] Миграция применяется на чистой БД (`migrate deploy`).  
- [ ] Уникальный лайк `(postId, userId)` не даёт дубликатов.  
- [ ] Каскадное удаление: удаление поста удаляет лайки и комментарии.  
- [ ] Удаление пользователя — поведение FK (Cascade или Restrict — как задано в схеме).  
- [ ] Таблица просмотров: повторный заход тем же пользователем не увеличивает `uniqueViewCount`.

### API

- [ ] Гость: `GET` список и деталь статьи — 200.  
- [ ] Гость: `POST` пост / лайк / комментарий — 401.  
- [ ] Авторизованный: создание поста — 201, пост появляется в списке.  
- [ ] Чужой пост: обычный пользователь `PUT`/`DELETE` — 403 или 404; **`moderator`** — 200 при валидном теле.  
- [ ] **Свой** комментарий: автор `PUT`/`DELETE` — 200.  
- [ ] Чужой комментарий: только **модератор** может `PUT`/`DELETE`.  
- [ ] Просмотр: после первого `GET` детали залогиненным — `uniqueViewCount` увеличился на 1; повторный запрос — без изменения.
- [ ] Лайк → счётчик увеличивается; повторный лайк — согласованное поведение (409 или no-op).  
- [ ] Снятие лайка — счётчик уменьшается.  
- [ ] Комментарий привязан к посту и автору; гость видит комментарии в `GET` детали.  
- [ ] Пагинация списка постов и (если есть) комментариев.  
- [ ] Невалидное тело — 400 с понятным `message`.

### Фронтенд

- [ ] Без входа: лента и статья открываются, лайк/коммент/«написать» ведут на вход или показывают подсказку.  
- [ ] После входа: можно опубликовать статью, лайкнуть, прокомментировать.  
- [ ] Редактирование/удаление **своей** статьи и **своих** комментариев; **модератор** — чужой пост и чужие комментарии.  
- [ ] На карточке/странице статьи отображается **число уникальных просмотров** (как в API).  
- [ ] Обложка загружается и отображается.  
- [ ] Навигация: пункт «Лента» ведёт на список.  
- [ ] Ошибки сети / 500 — пользователь видит сообщение (toast или текст на странице).

### Регрессия

- [ ] Существующие разделы (объявления, чат, объекты) не ломаются.  
- [ ] Сборка Docker / деплой: новые миграции попадают в образ и выполняются по инструкции README.

## 11. Решения по продукту (зафиксировано)

1. **URL фронтенда:** **`/lenta`** (маршруты `/lenta`, `/lenta/[id]`, `/lenta/new`, `/lenta/[id]/edit`). REST-префикс на бэкенде может остаться `/api/feed` или быть переименован в `/api/lenta` — на усмотрение реализации, главное — не смешивать пути страниц.

2. **URL статьи (v1 vs позже):** на **первом релизе** открытие по **`id` (UUID)** — маршрут `/lenta/[id]`, без обязательного slug в URL. **Позже** — добавить человекочитаемый **slug** в адрес (поле в БД, генерация, уникальность, редиректы при смене заголовка — по отдельной задаче).

3. **Модератор:** **только** роль **`moderator`** может **удалять и редактировать чужие** посты и комментарии.

4. **Свой контент у всех пользователей:** любой авторизованный пользователь может **редактировать и удалять только свои** статьи **и** **только свои** комментарии; чужой контент трогает **только** модератор (п. 3). Это **обязательно в v1** (API `PUT`/`DELETE` + UI).

5. **Тело статьи:** формат **Markdown** (ввод в форме, хранение в `body`, на фронте — рендер + **санитизация**).

6. **Просмотры:** счётчик **нужен**; метрика — **количество уникальных пользователей** (реализация через таблицу `FeedPostView` и `@@unique([postId, viewerId])`, учёт только **авторизованных** зрителей, см. §3.1).

### Справка (ранее обсуждавшиеся варианты)

П.п. 2, 4, 5 ранее были «открытыми»; варианты (UUID vs slug, правки своих комментариев, plain text vs Markdown) сведены в зафиксированные пункты выше.

## 12. Итог

План опирается на текущий стек (**Fastify + Prisma + PostgreSQL + Next.js + Zustand + JWT**). Минимальный объём для MVP: **таблицы пост / лайк / комментарий / просмотр (уникальный пользователь)**, **REST** (например префикс `/api/feed`), страницы **`/lenta`** с **`/lenta/[id]` по UUID**, **тело статьи в Markdown** (с санитизацией), **пользователь правит и удаляет только свои посты и комментарии**, **модератор** — чужой контент. Остальное (**slug в URL**, уведомления, черновики, учёт гостевых просмотров) — итеративно по разделам 7, 9 и 11.

## 13. Пошаговые промпты для реализации (для ИИ и человека)

**Как пользоваться:** выполняй шаги **по порядку** (каждый шаг опирается на предыдущие). В чат с ассистентом вставляй блок **«Промпт»** целиком; при необходимости добавь: «Следуй `BLOG.md` в корне репозитория, не выходи за рамки v1 из п. 11». После шага сверься с **«Готово, когда»** и с чек-листом **§10**. Если идёшь по этим шагам, фразу в шапке документа про «код пока не меняется» для задачи «Лента» можно игнорировать.

**Важно:** в одном шаге не смешивай несвязанные крупные фичи; если контекст чата переполнен — начни новый чат с кратким резюме «сделано до шага N» и ссылкой на изменённые файлы.

### Шаг 0 — контекст (опционально)
